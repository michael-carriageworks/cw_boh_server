Infrastructure & Security Brief

Back-of-House Technician Dashboard

A plain-language overview of how the dashboard is built, what information it uses, and the security considerations relevant to an IT review.

Status: Prototype (in testing) Owner: Carriageworks (Project manager: Michael) Prepared: 9 July 2026 Live at: cw-boh-server.vercel.app
In one line
A shared, live schedule board showing which technicians are rostered to which events across the precinct each week.
Data used
Event schedule from Smartsheet and staff rosters from Deputy — both pulled read-only.
Where it runs
Hosted database (Supabase) + web app (Vercel), with an automated sync job on GitHub. No on-premise servers.
Data sensitivity
Event schedules and staff roster details (names, shift times, roles). No financial, payment, or customer data.
Access control
A single shared login is being switched on. Per-person accounts are a planned upgrade.
01

What the dashboard is for

The dashboard is a shared screen for the Back-of-House (BOH) technical team. It runs on the display in the tech cage and is accessible online to all producers and senior staff, giving everyone the same live picture of the working week without anyone having to refresh or cross-check spreadsheets.

Its features point to its purpose:

  • A weekly Mon–Sun grid of events, colour-coded by activity type (bump in, rehearsal, performance, bump out, and so on), each showing its times, location, and the technicians rostered to it.
  • An auto-match between rostered shifts and events, so a technician's shift is attached to the right event automatically (marked with a "Deputy" tag), with anything ambiguous set aside for a producer to confirm.
  • A producer console for assigning technicians, confirming which location an event is in, adding ad-hoc events, and logging tasks.
  • "On now", "On later today", and "Tasks for today" summary panels for at-a-glance awareness.

This is a working prototype, currently in testing. Expect some visual/layout tweaks over the coming week as it's refined with the team. A few panels — weather, lock-up countdown, warden and evacuation status, and facilities notices — are clearly-labelled placeholders awaiting real data sources.

Live updates are pushed from the database straight to every open browser, so a change made on one device appears on the others within a second or two — no refresh needed.

Producer changes (assigning a technician, confirming a location, adding a task) are saved to the database and are deliberately preserved when the next automated sync runs, so edits are never overwritten.

03

What information is pulled

Only the fields needed for scheduling are read. Both sources are accessed read-only — the dashboard cannot create, change, or delete anything in Smartsheet or Deputy.

SourceWhat is readAccess
Smartsheet
event schedule		 A single report ("Precinct Ops Agenda"): project name, activity, activity start/end times, date, location, notes, and expected headcount. Read-only (data is only ever retrieved, never written back).
Deputy
staff rosters		 Rostered shifts for the 16 BOH technical role-locations only: employee name, shift times, and shift notes. Non-BOH areas are filtered out. Read-only (rosters are only retrieved, never modified).

The two are matched by project name, date, and overlapping times to work out which technician is on which event. Anything that can't be matched confidently is flagged for a producer to resolve, rather than guessed.

05

Security considerations

A candid summary for review — both the reassuring points and the items worth a decision.

Strength

Read-only to source systems

The sync only ever retrieves from Smartsheet and Deputy. It has no capability to alter events or rosters, so the dashboard cannot corrupt the systems it draws from.

Strength

No credentials in the code

All tokens and passwords live in encrypted platform settings. The public code repository was checked and contains no secrets.

Strength

Limited, non-sensitive data

Scope is event schedules and technical rosters. No financial, payment, ticketing, or customer data is involved.

Consideration

Data residency — Mumbai

The database currently sits in AWS Mumbai (India). Rosters include staff names (personal information). If Australian data-residency policy requires it, the database can be relocated to AWS Sydney with a short migration.

Consideration

Shared login (for now)

Access is being gated by a single shared username/password. There are no per-person accounts yet, so individual access can't be revoked separately. Per-user accounts are the recommended next step before wider rollout.

Consideration

Smartsheet account & permissions

The event data is currently pulled using the producer Smartsheet account — an editor-level account with access to all of the producers' Smartsheet files, which is far broader than the dashboard needs. Recommended: create a separate Smartsheet account used only by the dashboard, give it read-only permission, and share just the single "Precinct Ops Agenda" report with it. That way the dashboard can see exactly what it needs — and nothing else in Smartsheet.

Consideration

Public source code

The code is publicly viewable (which keeps the automated sync free). It exposes no credentials, but IT should be comfortable with the application logic being visible.

07

Room to grow — other departments

The dashboard is currently focused on Back-of-House because the roster feed is filtered to the 16 BOH technical roles. The underlying design is department-agnostic, though.

If Front-of-House felt this would be a useful tool for them, it can scale with a configuration change rather than a rebuild: the roster filter is simply pointed at the FOH roles instead, and FOH staff rosters flow in in place of the current BOH-focused view. The event schedule (Smartsheet) is already shared across the precinct, so no new data source is needed. Each department could ultimately have its own filtered view drawn from the same foundation.
